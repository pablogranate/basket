<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| app process → auth Postgres | The `AUTH_DATABASE_URL` connection string (server secret) crosses here |
| client bundle ↔ server modules | `auth-client.ts` must stay server-side only |
| operator → `.env.local` / container | Secrets and the running DB are introduced here |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-05 | Information Disclosure | Connection string leaking into a client bundle | mitigate | `src/lib/db/auth-client.ts` line 1 is `import "server-only";` (build-time guard; precedent `admin.ts`). The connection is only ever read via `appEnv` server-side. |
| T-01-06 | Spoofing / Tampering | Auth client accidentally reusing the Supabase connection | mitigate | The client imports only `@/lib/env` (`authDatabaseUrl`) and `@/lib/auth/schema`; acceptance criteria forbid any `@/lib/supabase/*` import (two-independent-connections rule). |
| T-01-07 | Information Disclosure | `AUTH_DATABASE_URL` / `AUTH_DB_PASSWORD` committed to git | mitigate | Set only in `.env.local`, which is gitignored (`.env*`, verified). The checkpoint instructs the operator never to commit them. |
| T-01-08 | Tampering | Applying the migration to the wrong (domain) DB | mitigate | Apply is `psql "$AUTH_DATABASE_URL" -f drizzle/auth/0000_*.sql` against the dedicated 5433 `basket_auth` container; `\dt` verification confirms only `auth_*` tables. No `drizzle-kit push` (D-09). |
