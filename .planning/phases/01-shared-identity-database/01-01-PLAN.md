<objective>
Produce the build-time artifacts for the shared `basket_auth` identity store: the Drizzle `pgTable` schema (canonical `auth_*` names with all admin-plugin columns baked in per D-07), an isolated auth-only drizzle-kit config, the committed SQL migration generated from that schema, the env wiring for `AUTH_DATABASE_URL`, and the docker-compose Postgres service for the dedicated `basket_auth` database (D-01). No Better Auth instance, no login methods, no app wiring — that is Phase 3.

Purpose: Establish the schema-of-record and the deploy artifacts so the shared store never needs a second migration when Phase 3 adds login methods (D-07), and so auth migrations stay fully isolated from domain data (D-11).
Output: `src/lib/auth/schema.ts`, `drizzle.auth.config.ts`, `docker-compose.yml`, modified `src/lib/env.ts`, a `db:auth:generate` package.json script, and the generated+committed `drizzle/auth/0000_*.sql` + `drizzle/auth/meta/`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| dev machine → git repo | Committed artifacts (config, schema, SQL, compose) must not carry secrets |
| app process → auth Postgres | A connection string (server secret) crosses here; consumed by the server-only client in Plan 02 |
| compose env → container | The Postgres superuser password is injected at container start |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Information Disclosure | `AUTH_DATABASE_URL` connection string | mitigate | Surfaced only via `appEnv` (server-read singleton); never inlined; the consuming client (Plan 02) is `import "server-only"`. `.env*` is gitignored (verified) so the URL is never committed. |
| T-01-02 | Information Disclosure / Tampering | Postgres superuser password in `docker-compose.yml` | mitigate | Compose uses `${AUTH_DB_PASSWORD:?set AUTH_DB_PASSWORD}` — secret comes from env and the container fails to start if unset; the password is never hard-coded in the committed file. |
| T-01-03 | Tampering | Auth migration accidentally targeting the Supabase/domain DB | mitigate | Isolated `drizzle.auth.config.ts` with `out: "./drizzle/auth"` and a dedicated `AUTH_DATABASE_URL`; Task 3 asserts `supabase/migrations/` is untouched. `drizzle-kit push`/Better Auth `migrate` forbidden (D-09) — no dev→DB write path. |
| T-01-04 | Elevation of Privilege | Weak/default Postgres password on the shared identity store | accept (Phase 1) / transfer to operator | The fail-fast `:?` syntax forces the operator to supply a password; strength is the operator's responsibility at provisioning. No auth logic runs in Phase 1. |
| T-01-SC | Tampering | npm/pnpm package installs | n/a | All four packages (`better-auth`, `drizzle-orm`, `drizzle-kit`, `postgres`) already in package.json and `[OK]` per RESEARCH.md Package Legitimacy Audit (slopcheck clean). No install task in this phase. |
</threat_model>
